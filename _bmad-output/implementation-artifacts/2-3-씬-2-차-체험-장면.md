# Story 2.3: 씬 2 — 차(車) 체험 장면

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want to be introduced to the Cha piece, watch a demonstration, move it myself, and receive Gyebaek's dialogue after my move,
so that I learn the Cha's movement through a guided experience embedded in the narrative.

## Acceptance Criteria

1. **Given** `state.scene === 'cha'` and `state.phase === 'intro'`
   **When** ExperienceCha가 렌더됨
   **Then** 차 기물이 등장하고, 황산벌에서 차의 역할("지휘하는 의지")을 설명하는 서사 텍스트가 표시됨

2. **Given** `state.phase === 'demo'`
   **When** 시연 단계에 진입
   **Then** 차 기물이 사전 스크립트된 경로를 따라 직선 이동하여 가로/세로 이동 규칙을 시각적으로 보여줌 (플레이어 입력 없음)

3. **Given** `state.phase === 'play'`
   **When** 플레이어가 조작권을 가짐
   **Then** 차 기물은 Story 1.5의 규칙으로 선택·이동 가능하며, 플레이어가 유효한 이동을 최소 1회 완료해야 다음 단계로 진행됨

4. **Given** 플레이어가 유효한 차 이동을 완료하여 `state.phase`가 `'dialogue'`로 전환됨
   **When** dialogue 단계가 렌더됨
   **Then** 계백의 대사 텍스트가 표시되며, 전투 맥락에서 차의 움직임을 언급함 (FR15)

5. **Given** dialogue가 표시됨
   **When** `useEffect`가 `state.phase === 'dialogue'`를 감지
   **Then** Epic 3에서 `audioManager.play(...)`를 호출할 훅 포인트가 코드에 존재함(현재는 실제 재생 없음)

6. **Given** 플레이어가 대사를 읽고 "계속하기"를 트리거
   **When** 액션이 발생
   **Then** `dispatch({ type: 'NEXT_SCENE' })`가 호출되어 `state.scene === 'ma'`로 전환됨

## Dev Notes

### 현재 구현 상태 — ExperienceCha.tsx (Story 2.1에서 생성된 스텁, 지금 UPDATE 대상)

```tsx
// src/scenes/ExperienceCha.tsx (현재)
import Board from '../board/Board'
import PiecesLayer from '../board/PiecesLayer'
import { useGameState, useGameDispatch } from '../state/GameContext'
import styles from './Scene.module.css'

export default function ExperienceCha() {
  const { phase } = useGameState()
  const dispatch = useGameDispatch()
  return (
    <div data-scene="cha" data-phase={phase ?? undefined} className={styles.scene}>
      <div className={styles.boardContainer}>
        <Board />
        <PiecesLayer />
      </div>
      {/* Story 2.3에서 intro/demo/play/dialogue 컨텐츠 추가 */}
      <button onClick={() => dispatch({ type: 'NEXT_SCENE' })}>계속하기</button>
    </div>
  )
}
```

`data-scene="cha"`, `data-phase={phase}`, Board+PiecesLayer 렌더는 **이미 완전히 동작함** — 그대로 유지. 무조건 `dispatch(NEXT_SCENE)`하던 하단 버튼만 phase 조건부 렌더로 교체 대상 (intro에서만 SET_PHASE, dialogue에서만 NEXT_SCENE).

### MOVE_PIECE는 `selectedPiece` 기반 — payload에 pieceId 없음 (가장 중요한 함정)

`gameReducer.ts`의 `MOVE_PIECE` 액션은 `{ type: 'MOVE_PIECE'; to: Position }` — 이동할 기물을 지정하지 않고 **현재 `state.selectedPiece`를 이동시킨다**. demo 단계에서 차를 스크립트로 이동시키려면 새 액션을 만들지 말고:

```typescript
dispatch({ type: 'SELECT_PIECE', id: 'cha' })
// ...setTimeout 이후...
dispatch({ type: 'MOVE_PIECE', to: DEMO_TARGET_CHA })
```

두 단계로 dispatch해야 한다. `MOVE_PIECE`는 이동 후 자동으로 `selectedPiece: null`로 리셋하므로 별도 DESELECT 불필요. `getValidMoves`가 실제로 그 경로를 유효하다고 판단하지 않으면(reducer가 `isLegal` 체크 후 조용히 무시) 데모가 아무 동작도 하지 않으므로 **목표 좌표는 반드시 실제 유효 이동이어야 한다.**

### DEMO_TARGET_CHA 좌표 계산 — 다른 기물 초기 위치를 건드리지 않도록 선택

`board`는 씬 전환 시 초기화되지 않는다(`NEXT_SCENE`은 `board`를 그대로 유지 — `gameReducer.ts` 참조). 즉 이 스토리에서 차를 이동시키면 그 위치가 Ma/Po/Jol 체험 씬(Story 2.4~2.6, 아직 미구현)까지 그대로 이어진다. `INITIAL_POSITIONS`는 `cha:(row5,col0)`, `ma:(row7,col2)`, `po:(row2,col4)`, `jol:(row5,col4)`.

`getChaValidMoves((5,0), board)` 유효 이동:
- 우측(row5): (5,1),(5,2),(5,3) — col4의 jol에 막힘
- 좌측: 없음(경계)
- 하단(col0): (6,0)~(9,0)
- 상단(col0): (4,0)~(0,0)

**`DEMO_TARGET_CHA = { row: 2, col: 0 }`로 고정** — col0 수직 이동만 사용해 ma(col2)/po(col4)/jol(col4) 초기 위치와 절대 겹치지 않는다. play 단계에서 플레이어가 다시 이동할 여지도 col0 상하 방향으로 충분히 남는다 (재계산 시 (3,0)~(9,0), (1,0),(0,0) 등 유효).

### play → dialogue 전환 감지 패턴 (Story 2.4~2.6 재사용 대상)

```tsx
const playStartPos = useRef<Position | null>(null)

useEffect(() => {
  if (phase !== 'play') { playStartPos.current = null; return }
  const pos = findPiecePosition('cha', board)
  if (!playStartPos.current) { playStartPos.current = pos; return }  // play 진입 시점 위치 캡처
  if (pos && (pos.row !== playStartPos.current.row || pos.col !== playStartPos.current.col)) {
    dispatch({ type: 'COMPLETE_PIECE', id: 'cha' })
    dispatch({ type: 'SET_PHASE', phase: 'dialogue' })
  }
}, [phase, board, dispatch])
```

`COMPLETE_PIECE`는 `gameReducer.ts`에 이미 구현되어 있으나 지금까지 어느 스토리도 dispatch하지 않았다(Story 1.2 리뷰 노트: "게임 로직 레이어(Story 2.x)에서 처리 예정" — 바로 이 스토리가 그 대상). 다른 기물(ma/po/jol)을 이 phase 도중 움직여도(현재 PiecesLayer는 phase 게이팅이 없어 기술적으로 가능) 이 effect는 오직 `cha`의 위치 변화만 감지하므로 오작동하지 않는다.

### PiecesLayer는 phase로 게이팅하지 않는다 (의도적 결정, 변경 금지)

`PiecesLayer.tsx`는 현재 모든 기물을 항상 클릭 가능하게 렌더한다(phase 무관). Story 1.5~1.8의 테스트(`story-1-6/1-7/1-8-*.spec.ts`)가 **`cha` 씬에 진입한 상태에서 ma/po/jol을 직접 클릭·이동**시켜 각 기물의 룰을 검증한다 — 이 테스트들은 scene='cha'에서 실행되지만 특정 phase를 가정하지 않는다. 이번 스토리에서 "play 단계가 아니면 다른 기물 선택 금지" 같은 게이팅을 추가하면 1.5~1.8 테스트 12개 이상이 즉시 깨진다. **PiecesLayer.tsx는 수정하지 않는다** — 플레이어가 intro/demo/dialogue 중 다른 기물을 클릭할 수 있는 것은 알려진 POC 스코프 한계로 남겨두고 Story 3.4(비주얼/인터랙션 정제)로 defer.

```typescript
}, [state.phase, state.scene])
```
하지만 `audioManager.play(key: AudioKey)`의 `AudioKey`는 `audioKeys.ts`의 `AUDIO_KEYS` 리터럴 유니온으로 엄격 타입되어 있고, 현재 `'cha-dialogue'` 같은 대사 전용 키가 정의되어 있지 않다 — 템플릿 문자열을 그대로 `play()`에 넘기면 TypeScript strict 컴파일 에러가 난다. 이 스토리는 **실제 `audioManager.play()` 호출은 하지 않는다** — AC5는 "훅 포인트가 존재"만 요구하므로 주석으로 대체:

```tsx
useEffect(() => {
  if (phase === 'dialogue') {
    // Story 3.2: audioManager.play(AUDIO_KEYS.DIALOGUE_CHA) 예정
    // audioKeys.ts에 대사 전용 키 아직 없음 — Story 3.2에서 AUDIO_KEYS 확장 필요(동적 템플릿 문자열 대신 명시적 키 권장)
  }
}, [phase])
```
Story 3.2 착수 시 `AUDIO_KEYS`에 `DIALOGUE_CHA`/`DIALOGUE_MA`/`DIALOGUE_PO`/`DIALOGUE_JOL` 등 명시적 키 추가를 권장(architecture.md의 동적 템플릿 문자열 패턴은 타입 안전성과 충돌하므로 따르지 않는다).

### Story 2.1 테스트 리그레션 — 구체적 수정 코드

`tests/story-2-1-scene-transition.spec.ts`의 AC2/AC4는 각 씬에서 "계속하기" 버튼 1회 클릭이 즉시 다음 씬으로 이어진다고 가정한다. cha 씬은 이제 그렇지 않다(intro→demo→play→dialogue 4단계 필요). 헬퍼 함수를 추가하고 cha 구간만 교체한다:

```typescript
async function completeChaExperience(page: Page) {
  // intro → demo
  await page.locator('[data-scene="cha"] button:has-text("계속하기")').click()
  // demo → play (자동, DEMO_DELAY_MS 경과 후)
  await page.waitForSelector('[data-scene="cha"][data-phase="play"]', { timeout: 5000 })
  // play: 유효한 이동 1회 수행
  await page.locator('[aria-label="기물 cha"]').click()
  await page.getByRole('button', { name: /로 이동/ }).first().click()
  // play → dialogue (자동)
  await page.waitForSelector('[data-scene="cha"][data-phase="dialogue"]', { timeout: 5000 })
  // dialogue → ma
  await page.locator('[data-scene="cha"] button:has-text("계속하기")').click()
}
```

`AC2`(6단계 전체 순회)와 `AC4`(ending 도달) 테스트에서 `cha` 구간을 `SCENES_WITH_NEXT`의 단순 루프 대신 `await completeChaExperience(page)` 호출로 교체한다. `ma`/`po`/`jol`은 여전히 Story 2.1의 단순 스텁이므로 루프 로직 변경 없음. `AC1`, `AC3`, `AC5`는 opening→cha 진입 단계만 다루므로 영향 없음.

### 힌트 원 셀렉터 — `getByRole('button', { name: /로 이동/ })` 사용 이유

`PiecesLayer.tsx`의 힌트 `<circle>`은 `role="button"` + `aria-label="(row, col)로 이동"`을 가진다. Board.tsx에는 role="button" 요소가 없으므로 이 셀렉터는 스코프 내에서 안전하게 유일한 매치를 보장한다. 좌표 하드코딩 대신 `.first()`로 임의의 유효 이동 칸을 클릭하면 좌표 계산 실수를 피할 수 있다.

### Project Context Rules

- **dispatch만 사용** — phase 전환은 `SET_PHASE`, 기물 이동은 `SELECT_PIECE`+`MOVE_PIECE` 조합만 사용. `useState`로 로컬 UI 상태를 만들지 않는다(단, `playStartPos` 같은 렌더에 영향 없는 ref는 허용 — GameState에 속하지 않는 effect 내부 추적용)
- **CSS Modules** — 신규 스타일은 기존 `Scene.module.css` 재사용, 인라인 스타일 금지
- **TypeScript strict** — `CHA_INTRO`/`CHA_DIALOGUE`는 `as const`
- **매직 넘버 금지** — `DEMO_TARGET_CHA`, `DEMO_DELAY_MS`는 명명된 상수로 파일 상단에 선언, 다른 곳에 하드코딩 금지
- **오디오는 audioManager.play(key)만** — 이번 스토리는 실제 호출 없음(위 "오디오 훅 포인트" 참조), Howler 직접 호출 절대 금지
