# Story 2.4: 씬 3 — 마(馬) 체험 장면

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want to be introduced to the Ma piece and experience its L-shaped movement through the same guided structure as the Cha scene,
so that I learn the horse's distinctive movement within the continuing narrative.

## Acceptance Criteria

1. **Given** `state.scene === 'ma'` and `state.phase === 'intro'`
   **When** ExperienceMa가 렌더됨
   **Then** 마 기물이 등장하고, 황산벌에서 마의 역할("전령")을 설명하는 서사 텍스트가 표시됨

2. **Given** `state.phase === 'demo'`
   **When** 시연 단계에 진입
   **Then** 마 기물이 사전 스크립트된 L자 이동(직선 1칸 + 대각선 1칸)을 1회 실행하여 이동 패턴을 시각적으로 보여줌 (플레이어 입력 없음)

3. **Given** `state.phase === 'play'`
   **When** 플레이어가 조작권을 가짐
   **Then** 마 기물은 Story 1.6의 규칙으로 선택·이동 가능하며, 플레이어가 유효한 이동을 최소 1회 완료해야 다음 단계로 진행됨

4. **Given** 플레이어가 유효한 마 이동을 완료하여 `state.phase`가 `'dialogue'`로 전환됨
   **When** dialogue 단계가 렌더됨
   **Then** 계백의 대사 텍스트가 표시되며, 전투 맥락에서 마를 언급함 (FR15)

5. **Given** 플레이어가 대사를 읽고 "계속하기"를 트리거
   **When** 액션이 발생
   **Then** `dispatch({ type: 'NEXT_SCENE' })`가 호출되어 `state.scene === 'po'`로 전환됨

  const playStartPos = useRef<Position | null>(null)

### 데모 목표를 동적으로 계산해야 하는 이유 (Story 2.3과의 가장 중요한 차이점)

Story 2.3은 `DEMO_TARGET_CHA`를 파일 상단에 **하드코딩된 절대 좌표**로 선언했다. 이게 안전했던 이유는 차 씬이 게임 전체에서 **최초로 board가 상호작용 가능해지는 씬**이었기 때문 — opening 씬은 Board/PiecesLayer를 렌더하지 않으므로, 차의 데모가 실행되는 시점에 어떤 기물도 초기 위치(`INITIAL_POSITIONS`)에서 벗어나 있을 수 없었다.

**마 씬은 다르다.** `PiecesLayer.tsx`는 phase나 씬에 관계없이 항상 모든 기물을 클릭 가능하게 렌더한다(Story 2.3에서 의도적으로 유지한 결정 — 게이팅 시 Story 1.5~1.8 테스트가 깨짐). 즉 플레이어가 **차 씬(intro/demo/play/dialogue 어느 단계에서든)** 동안 마·포·졸을 자유롭게 클릭해 이동시켰을 수 있다. 마 씬에 진입했을 때 마가 여전히 `INITIAL_POSITIONS.ma = { row: 7, col: 2 }`에 있다는 보장이 없다.

**해결책:** 데모 목표를 하드코딩하지 않고 데모 실행 시점에 `getValidMoves('ma', currentPos, board)`로 동적 계산한다. 다행히 `getMaValidMoves`(및 `cha`/`po`/`jol` 룰 엔진 전부)는 **점유된 칸을 이미 목적지 후보에서 제외**한다(`ma.ts:28` `if (board[tr][tc] !== null) continue`) — 이는 Story 1.5 리뷰에서 명시적으로 요구된 계약("다른 기물 룰 구현 시 해당 기물의 rule engine이 occupied 칸을 반환하지 않는지 확인 필요")이 마 구현 시 실제로 충족된 것이다. 따라서 동적으로 얻은 첫 번째 유효 목적지는 **다른 기물이 어디에 있든 항상 안전**하다 — 별도의 충돌 회피 계산이 필요 없다.

```tsx
// ExperienceMa.tsx
const boardRef = useRef(board)
boardRef.current = board // 매 렌더마다 최신값 유지, effect 의존성에는 넣지 않음

useEffect(() => {
  if (phase !== 'demo') return
  const timer = setTimeout(() => {
    const currentBoard = boardRef.current
    const currentPos = findPiecePosition('ma', currentBoard)
    const demoTarget = currentPos ? getValidMoves('ma', currentPos, currentBoard)[0] : undefined
    if (currentPos && demoTarget) {
      dispatch({ type: 'SELECT_PIECE', id: 'ma' })
      dispatch({ type: 'MOVE_PIECE', to: demoTarget })
    }
    dispatch({ type: 'SET_PHASE', phase: 'play' })
  }, DEMO_DELAY_MS)
  return () => clearTimeout(timer)
}, [phase, dispatch])
```

**`board`를 이 effect의 의존성 배열에 넣지 말 것.** 만약 넣으면 대기 중 다른 기물이 움직일 때마다(=`board` 변경) effect가 cleanup+재실행되어 타이머가 계속 리셋되고, 플레이어가 아무 기물이나 계속 클릭하면 데모가 영원히 시작되지 않는 상태에 빠질 수 있다. `boardRef`로 우회해 "타이머는 phase 진입 시 한 번만 예약하되, 콜백이 실제로 실행되는 순간의 최신 board를 읽는다"를 달성한다. (이 패턴은 Story 2.3에는 필요 없었지만 — 하드코딩된 좌표라 board를 읽을 필요가 없었다 — 2.5/2.6에서도 그대로 재사용할 것을 권장. Story 2.3의 데모를 이 패턴으로 소급 정리하는 것도 고려할 만하지만 이번 스토리 범위는 아니다.)

`currentPos`가 `null`이거나(마가 보드에서 사라진 경우 — 이론상으로만 가능, deferred-work.md의 Story 2.3 항목 참조) `getValidMoves`가 빈 배열을 반환하는 극단적 경우에도 `MOVE_PIECE`를 건너뛰고 `SET_PHASE('play')`는 그대로 진행한다 — 게임이 멈추지 않도록 조용히 무시(Project Context Rules의 에러 처리 원칙과 동일).

**일반적인(교란 없는) 경우 실제로 어떤 이동이 나오는지 참고:** 마가 `INITIAL_POSITIONS.ma = (row:7,col:2)`에 그대로 있다면 `MA_STEPS`의 첫 방향("상")이 `(row:5,col:1)`을 반환한다(경로 확인: leg `(6,2)` 비어있음, 목적지 `(5,1)` 비어있음) — 시각적으로 자연스러운 데모가 된다. 하지만 코드는 이 값을 하드코딩하지 않는다.

### getValidMoves import 경로

```tsx
import { getValidMoves } from '../pieces/rules/ruleEngine'
```
