# Story 2.5: 씬 4 — 포(砲) 체험 장면

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want to be introduced to the Po piece and experience its unique jump mechanic through the guided scene structure,
so that I understand the cannon's distinctive rule before trying it myself.

## Acceptance Criteria

1. **Given** `state.scene === 'po'` and `state.phase === 'intro'`
   **When** ExperiencePo가 렌더됨
   **Then** 포 기물이 등장하고, 반드시 기물 1개를 넘어야 이동/공격할 수 있다는 규칙을 강조하는 서사 텍스트가 표시됨

2. **Given** `state.phase === 'demo'`
   **When** 시연 단계에 진입
   **Then** 포 기물이 경유 기물 1개를 넘는 사전 스크립트된 점프를 1회 실행하여 매커니즘을 명확히 시각적으로 보여줌 (플레이어 입력 없음)

3. **Given** `state.phase === 'play'`
   **When** 플레이어가 조작권을 가짐
   **Then** 포 기물은 Story 1.7의 규칙(3가지 엣지 케이스 포함)으로 선택·이동 가능하며, 플레이어가 유효한 이동을 최소 1회 완료해야 다음 단계로 진행됨

4. **Given** 플레이어가 유효한 포 이동을 완료하여 `state.phase`가 `'dialogue'`로 전환됨
   **When** dialogue 단계가 렌더됨
   **Then** 계백의 대사 텍스트가 표시되며, 전투 맥락에서 포를 언급함 (FR15)

5. **Given** 플레이어가 대사를 읽고 "계속하기"를 트리거
   **When** 액션이 발생
   **Then** `dispatch({ type: 'NEXT_SCENE' })`가 호출되어 `state.scene === 'jol'`로 전환됨

### ExperienceMa.tsx의 코드 리뷰 이후 최종본 — 이번 스토리가 재사용할 정확한 패턴

**중요:** Story 2.4는 최초 구현 시 `boardRef`로 데모 타이머의 fire-time에 board를 다시 읽는 방식을 썼다가, 코드 리뷰에서 "대기 중 플레이어가 기물을 옮기면 그 새 위치에서 또 유효한 이동을 찾아 이중으로 실행된다"는 결함이 발견되어 패치되었다. **이번 스토리는 처음부터 패치된 최종 버전을 따른다** — `boardRef`를 아예 만들지 않는다:

```tsx
// src/scenes/ExperienceMa.tsx (코드 리뷰 반영 후 최종본 — 이 구조를 그대로 따라간다)
useEffect(() => {
  if (phase !== 'demo') return
  const currentPos = findPiecePosition('ma', board)
  const demoTarget = currentPos ? getValidMoves('ma', currentPos, board)[0] : undefined
  const timer = setTimeout(() => {
    if (currentPos && demoTarget) {
      dispatch({ type: 'SELECT_PIECE', id: 'ma' })
      dispatch({ type: 'MOVE_PIECE', to: demoTarget })
    }
    dispatch({ type: 'SET_PHASE', phase: 'play' })
  }, DEMO_DELAY_MS)
  return () => clearTimeout(timer)
  // eslint-disable-next-line react-hooks/exhaustive-deps -- board는 demo 진입 시점 값만 필요, 재실행 시 타이머가 리셋되는 문제 방지
}, [phase, dispatch])
```

핵심: `currentPos`/`demoTarget`은 **effect 본문에서 (setTimeout 밖에서) 한 번만 계산**해 클로저에 고정한다. `board`는 effect 의존성 배열에 넣지 않는다(대기 중 board가 바뀔 때마다 effect가 재실행되면 타이머가 계속 리셋되는 문제 방지, `react-hooks/exhaustive-deps` 경고는 의도적으로 비활성화). play→dialogue 감지 패턴(`playStartPos` ref)도 `ExperienceMa.tsx`에서 그대로 복사, `pieceId`만 `'po'`로 교체.

### 포(砲)는 다른 기물과 근본적으로 다른 이동 메커니즘 — 데모 설계 시 주의

차/마는 "이동 가능한 칸이 있는가"만 걱정하면 됐지만, 포는 **"직선상에 경유 기물이 정확히 1개 있는 방향이 있는가"**가 있어야 데모/플레이가 가능하다(`po.ts` 참조 — Phase 1에서 경유 기물 탐색, 못 찾으면 그 방향은 스킵). 이는 차/마보다 "유효 이동 0개" 상황이 발생할 가능성이 구조적으로 더 높다:

- `INITIAL_POSITIONS`는 po(row:2,col:4)와 jol(row:5,col:4)을 같은 열(col4)에 배치해 **의도적으로 기본 점프 데모가 가능하도록 설계**되어 있다(`constants.ts` 주석 참조 — 단, 주석은 "Story 1.7/2.4"라고 적혀 있는데 이는 스토리 번호가 재편되기 전의 stale 주석이다, 실제로는 이번 스토리 2.5가 그 데모임).
- 하지만 이 얼라인먼트는 **차/마 씬 동안 플레이어가 po나 jol을 직접 움직였다면 깨질 수 있다**(PiecesLayer가 게이팅되지 않는다는 Story 2.3의 결정이 그대로 적용됨). 마의 최종 위치는 Story 2.4에서 동적으로 계산되므로 이 시점엔 이미 예측 불가능한 위치에 있을 수 있고, 차의 위치도 플레이어가 play 단계에서 어디로 이동시켰는지에 따라 다르다.
- 따라서 이번 스토리는 (Story 2.4와 동일하게) 데모 목표를 하드코딩하지 않고 `getValidMoves`로 동적 계산한다. **일반적인(교란 없는) 경우** `po.ts`의 `DIRECTIONS` 순서(상→하→좌→우)상 "상" 방향은 경유 기물이 없어 스킵되고, "하" 방향에서 jol을 경유 기물로 찾아 `(row:6,col:4)`가 첫 유효 목적지가 된다 — 참고용이며 코드에 하드코딩하지 않는다.
- `getValidMoves`가 빈 배열을 반환하는 경우(포가 어느 방향으로도 정확히 1개를 넘을 수 없는 상태) `demoTarget`이 `undefined`가 되어 데모 이동 없이 곧장 `play`로 넘어간다 — Story 2.3/2.4에서 이미 동일한 패턴으로 "defer" 처리된 케이스(소프트락 이론상 가능, 좁은 트리거)와 같은 원칙을 따른다. 포는 이 케이스에 도달할 확률이 차/마보다 높지만, 별도의 폴백 메커니즘(예: 강제로 경유 기물을 만드는 로직)을 새로 설계하는 것은 이번 스토리 범위를 벗어난다 — 발생 시 defer로 기록.

### Project Context Rules

- **dispatch만 사용** — phase 전환은 `SET_PHASE`, 기물 이동은 `SELECT_PIECE`+`MOVE_PIECE` 조합만 사용
- **CSS Modules** — 신규 스타일 없음(기존 `Scene.module.css`의 `.narrationText` 재사용), 인라인 스타일 금지
- **TypeScript strict** — `PO_INTRO`/`PO_DIALOGUE`는 `as const`
- **매직 넘버 금지** — `DEMO_DELAY_MS`는 명명된 상수로 선언(Ma와 동일하게 1000 권장, 일관성)
- **룰 엔진** — `getValidMoves(pieceId, pos, board)`만 사용, `getPoValidMoves` 직접 import 금지
- **PiecesLayer.tsx는 수정하지 않는다**
- **`boardRef` 패턴을 쓰지 않는다** — Story 2.4 코드 리뷰에서 폐기된 패턴, 데모 목표는 effect 시작 시점에 한 번만 계산
