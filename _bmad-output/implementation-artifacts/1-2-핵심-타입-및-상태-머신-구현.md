# Story 1.2: 핵심 타입 및 상태 머신 구현

Status: done

## Story

As a developer,
I want the core GameState types, gameReducer, ChapterConfig, and constants to be implemented,
so that all game state changes flow through a single type-safe reducer and all magic numbers are centralized.

## Acceptance Criteria

1. **Given** `state/gameReducer.ts` with initial scene `'opening'`  
   **When** `dispatch({ type: 'NEXT_SCENE' })` 가 반복 호출될 때  
   **Then** 씬이 순서대로 진행: `opening → cha → ma → po → jol → ending` (ending에서 멈춤)

2. **Given** `state/gameReducer.ts`  
   **When** `dispatch({ type: 'SELECT_PIECE', id: 'cha' })` 가 호출될 때  
   **Then** `state.selectedPiece === 'cha'`

3. **Given** `constants.ts`  
   **When** `PIECE_ANIMATION` 을 import할 때  
   **Then** `cha: { duration: 500, ease: 'ease-in' }`, `ma: { duration: 300, ease: 'ease-out' }`, `po: { duration: 150, ease: 'linear' }`, `jol: { duration: 400, ease: 'ease-in' }` 를 포함하며, 이 숫자들은 다른 어떤 파일에도 raw number로 존재하지 않음

4. **Given** `pieces/config/gyebaekChapter.ts`  
   **When** `GYEBAEK_CHAPTER` 을 import할 때  
   **Then** `cha`, `ma`, `po`, `jol` 각각에 대해 `id`, `symbol`, `animation` (PIECE_ANIMATION에서 참조), `sound` 키 (AUDIO_KEYS에서 참조), `moves` 함수 참조가 포함됨

5. **Given** TypeScript strict mode 활성화 상태  
   **When** 프로젝트가 컴파일될 때  
   **Then** `state/` 및 `pieces/config/` 에서 TypeScript 에러 0건

### GameState & Action 타입 (아키텍처 확정)

```typescript
// src/state/gameTypes.ts

import type { PieceId, Position } from '../types'

export type Scene = 'opening' | 'cha' | 'ma' | 'po' | 'jol' | 'ending'
export type ExperiencePhase = 'intro' | 'demo' | 'play' | 'dialogue'

export type GameState = {
  scene: Scene
  phase: ExperiencePhase | null   // 내러티브 장면(opening/ending)은 null
  selectedPiece: PieceId | null
  completedPieces: Set<PieceId>
}

export type Action =
  | { type: 'NEXT_SCENE' }
  | { type: 'SET_PHASE'; phase: ExperiencePhase }
  | { type: 'SELECT_PIECE'; id: PieceId }
  | { type: 'DESELECT_PIECE' }
  | { type: 'MOVE_PIECE'; to: Position }
  | { type: 'COMPLETE_PIECE'; id: PieceId }
```

### gameReducer 핵심 구현 패턴

```typescript
// src/state/gameReducer.ts

import type { GameState, Action, Scene } from './gameTypes'

export const SCENE_ORDER: readonly Scene[] = [
  'opening', 'cha', 'ma', 'po', 'jol', 'ending',
] as const

const NARRATIVE_SCENES = new Set<Scene>(['opening', 'ending'])

export const initialState: GameState = {
  scene: 'opening',
  phase: null,
  selectedPiece: null,
  completedPieces: new Set(),
}

export function gameReducer(state: GameState, action: Action): GameState {
  switch (action.type) {
    case 'NEXT_SCENE': {
      const idx = SCENE_ORDER.indexOf(state.scene)
      const next = SCENE_ORDER[Math.min(idx + 1, SCENE_ORDER.length - 1)]
      return {
        ...state,
        scene: next,
        phase: NARRATIVE_SCENES.has(next) ? null : 'intro',
        selectedPiece: null,
      }
    }
    case 'SET_PHASE':
      return { ...state, phase: action.phase }
    case 'SELECT_PIECE':
      return { ...state, selectedPiece: action.id }
    case 'DESELECT_PIECE':
      return { ...state, selectedPiece: null }
    case 'MOVE_PIECE':
      return { ...state, selectedPiece: null }  // 실제 위치 추적은 Story 1.4
    case 'COMPLETE_PIECE':
      return { ...state, completedPieces: new Set([...state.completedPieces, action.id]) }
    default:
      return state
  }
}
```

**중요:** `Set`은 spread 없이 직접 mutate하면 React 리렌더 감지가 안 됨. 반드시 `new Set([...prev, id])` 패턴 사용.

### GameContext 패턴 (State/Dispatch 분리)

State와 Dispatch를 분리된 Context로 제공 → Dispatch만 쓰는 컴포넌트가 state 변경으로 불필요하게 리렌더되지 않음.

```typescript
// src/state/GameContext.tsx
import { createContext, useContext, useReducer, type ReactNode } from 'react'
import { gameReducer, initialState } from './gameReducer'
import type { GameState, Action } from './gameTypes'

const GameStateCtx = createContext<GameState | null>(null)
const GameDispatchCtx = createContext<React.Dispatch<Action> | null>(null)

export function GameProvider({ children }: { children: ReactNode }) {
  const [state, dispatch] = useReducer(gameReducer, initialState)
  return (
    <GameStateCtx.Provider value={state}>
      <GameDispatchCtx.Provider value={dispatch}>
        {children}
      </GameDispatchCtx.Provider>
    </GameStateCtx.Provider>
  )
}

export function useGameState(): GameState {
  const ctx = useContext(GameStateCtx)
  if (!ctx) throw new Error('[GameContext] useGameState must be inside GameProvider')
  return ctx
}

export function useGameDispatch(): React.Dispatch<Action> {
  const ctx = useContext(GameDispatchCtx)
  if (!ctx) throw new Error('[GameContext] useGameDispatch must be inside GameProvider')
  return ctx
}
```

### 코드 리뷰에서 배운 점 (Story 1.1 → Story 1.2 적용)

- `strict: true`가 tsconfig.app.json에 추가됨 → `null` 체크 엄격. `GameState`의 nullable 필드 처리 주의
- Context value가 `null`일 경우 → hook에서 명확한 에러 메시지로 throw (non-null assertion `!` 금지)
- GameProvider 바깥에서 `useGameState()` 사용 시 런타임 에러 — 유용한 에러 메시지 필수
- `Set` 타입은 `strictNullChecks` 활성화 시 spread 패턴 필수

### TypeScript strict 주의사항

`NARRATIVE_SCENES.has(next)` 를 삼항 연산자에서 쓸 때 반환 타입이 정확한지 확인:
```typescript
phase: NARRATIVE_SCENES.has(next) ? null : 'intro' as ExperiencePhase
```
타입 추론이 잘 안 될 경우 명시적 타입 캐스팅 추가.

### Project Context Rules

- **GSAP**: 이 스토리에서는 GSAP 미사용. Story 1.3+에서 `useGSAP + contextSafe` 패턴 필수
- **Howler**: `audioManager.play(key: AudioKey)` 만 사용. gyebaekChapter.ts의 sound 필드는 AudioKey 타입 (`AUDIO_KEYS.SFX_CHA` 등)으로 설정

- **dispatch**: 이 스토리에서 GameContext 설정. 이후 모든 상태 변경은 `dispatch`로만
- **매직 넘버**: 500, 300, 150, 400 — `constants.ts`의 PIECE_ANIMATION에서만 참조
- **에러 처리**: Context null 체크 시 `throw new Error(...)` (에러 UI 노출 아님 — 이는 개발 시점 에러)
